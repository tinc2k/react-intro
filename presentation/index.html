http://facebook.github.io/flux/docs/overview.html - application architecture that Facebook uses for building client-side web apps - unidirectional data flow - more of a pattern rather than a formal framework - action creators — dispatcher helper methods — used to support a semantic API that describes all changes that are possible in the application - eschews MVC in favor of a unidirectional data flow: dependencies and cascading updates are difficult - various stores hold the application's data and business logic - stores update all the views that are affected - stores have no direct setter methods like setAsRead(), but instead have only a single way of getting new data into their self-contained world — the callback they register with the dispatcher. - The actions are simple objects containing the new data and an identifying type property. - actions are provided to the dispatcher in an _action creator_ method - stores register callbacks to the dispatcher - stores then emit a _change_ event to alert the controller-views - Application state is maintained only in the stores - balance the gain of simpler components against the complexity of multiple data updates flowing into the hierarchy at different points
